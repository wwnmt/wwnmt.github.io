Makefile的重要性

  Windows下编程的大佬们请忽略这篇短文，因为Windows的IDE像一名体贴细致的英式管家为程序员完成了Makefile的工作。如果在UNIX下进行一个开发项目，一个工程中有不计其数的源文件，每次修改代码后都要为每一个修改过的文件重新编译那是很麻烦的。从事计算机行业的人大都不喜欢麻烦。Makefile此时带来了“自动化编译”的好处(本初使用GNU项目组的make)。使用Makefile定义整个工程的编译规则，写好之后一个简单的make [all, clean, run…]命令就可以完成整个工程的编译。

  大三上学期的PA实验中，老师为我们写好了完整的Makefile文件让我们在实验中可以轻松的完成对代码的编译运行，而当时看到Makefile的内容后简直是一头雾水，就像看下面的例子一样。

初探Makefile

首先看一下Makefile的语法:
  target … : prerequisites …
     command
     …
  语法意思是想要得到target，需要执行command，而target依赖于prerequisites，当prerequisites中至少有一个文件比target新时，才执行command，make会自动比较目标和源文件的新旧程度，以免做不必要的重复性工作。下面是一个实例，2个.asm文件，4个.inc文件。

  Makefile可以定义变量来简化操作，有更直观的视觉效果，”=”用来定义变量，使用大写字母来命名变量，ASM，FLAGS,TARGET就是变量，在使用变量时需要使用$(变量名)，这点类似UNIX的shell命令使用方法。接下来的内容再结合上面的Makefile语法就很好理解了，command”$(ASM) $(SMFLAGS) -o $@ $<”还有两个神秘的符号”$@”和”$<”，这两个符号分别代表target，prerequisites的第一个项。所以那句command等价于：
    nasm -o boot.bin boot.asm
  注意，clean不是一个target文件，而是一个action，当没有prerequisites时，make不会去寻找依赖文件，而是直接执行command，执行”make clean”后，相当于执行”rm -f boot.bin loader.bin”将编译后的文件删除。
  接下来我们再往Makefile中补充几行。.PHONY也修改了，.PHONY表示后面的符号不是文件名而是一种action符号。all后跟着clean和everything,表明执行”make all”时，首先执行”make clean”，再执行”make everything”。

  还需要再注意的一点是Makefile的位置和所编译的源文件、头文件的目录位置关系。当Makefile的位置更换时，要让make通过Makefile找到源文件和头文件进行编译，需要在这些文件前加上准确的路径，以免找到了同名但错误的文件进行编译。
